# Bluesky（AT 协议）

Bluesky 是搭建在一套共同规则——**AT 协议（AT Protocol）**——之上的社交网络。实际目标很简单：让人们更容易使用社交应用，而不必完全绑死在某一家公司的服务器选择上。

与「一个平台 = 账号和内容只待在一个地方」不同，AT 协议把几件事拆开：

- **你在哪里发帖、存放身份与内容**（主页托管方 / home provider）
- **公开内容如何被分发**（分发层）
- **你的信息流体验如何拼装**（信息流 / 视图层）

这种拆分有助于生态做大：各家公司可以在产品体验上竞争，而底层协议仍让网络保持互通。

## 网络结构（通俗说法）

可以把系统想成三块：

1. **大本营（你发帖的地方）**
   - 账号数据和你的公开帖文，存在你所选的主页托管方那里。
   - 也就是说，「托管」和「写入」可以在不同提供商之间相对独立。

2. **广域分发（公开内容如何扩散）**
   - 当你公开发布时，更新可以被推送到更广的网络。
   - 这依赖帮助分发公开事件的服务。

3. **信息流体验（你真正看到什么）**
   - 时间线、排序、过滤、审核策略等，可以由信息流或视图提供方来产出。
   - 这是应用团队在个性化与产品体验上创新的空间。

### 简易示意图

```text
用户（发帖） ---> 大本营（账号 + 写入） ---> 分发 ---> 信息流/视图（用户所见）
                    (PDS)                      (relays)        (时间线/排序等)
```

## AT 协议与基于 HTTP 的万维网（一个有用类比）

AT 也可以被理解成**互联网式**的：**分层**、**开放规则**、**多种实现**并存，而不是由一家公司独占整条栈。

**类比成立之处**

- **分层架构。** 万维网叠了命名（DNS）、传输（TCP/TLS）、应用动词（HTTP）以及表现形式（HTML、JSON 等）。AT 则叠了**主页仓库（PDS）**、**公开分发（中继、广域事件流）**与**信息流与应用（视图层）**。两类设计都常常允许**改一层而不必推倒重来**。
- **协议而非单一产品。** HTTP 不是「Chrome 的私有协议」；AT 也不应被理解成某一应用的封闭后台 API。二者都追求**多家独立实现仍能互通**。
- **标识与解析。** URL 把主机与路径绑到资源上；AT 把**句柄与 DID** 绑到托管你**已签名仓库**的**主页服务器**。本质相近：**先解析名字，再访问提供权威（或约定）数据的一方**。
- **纸面开放、体验集中。** 流量与默认入口仍会向**大型托管方与头部应用**聚集——这与前文对 AT 的观察一致，也与**电子邮件、DNS、普通网站托管**的常态一致。

**类比需要收一收的地方**

- **HTTP 薄而通用；AT 厚且面向社交。** HTTP 描述如何请求资源；AT 则规定**记录类型、仓库、签名与删除语义**等，服务于**社交身份与公开数据**。AT 更接近 **HTTP 加上一套专门的 schema 与同步模型**，而不是又一版泛用文件传输。
- **交互形态。** 经典 Web 多为**无状态请求 / 响应**与**拉取**。AT 侧重**持续复制、事件流与仓库语义**，更接近信息流与同步，而非「打开一页即结束」。
- **信任与可迁移性。** 寻常 HTTPS 站点信任 **TLS、DNS 与站点运营方**。AT 额外强调**密码学仓库与可迁移身份**，因为**社交图**在运营方之间迁移，比拉取 HTML 要难得多。

**一句话：** AT 试图为**社交身份与公开记录**做的，类似于开放 Web 曾为**文档与服务**做的：**共享协议、分层角色、多款产品共存**——同时也重复出现**示意图上的去中心化**与**用户真实体感**之间的落差。

## 公开帖文与删除：有利于问责的一面

当用户删除一条公开帖文时，协议会更新其主页仓库，合规的应用会**在「官方图」里不再展示该帖**。但它**无法**收回已经落在中继、索引、存档或他人截图文件夹里的每一份副本——监听公开更新的基础设施也能看到**发生过一次删除**。

对**监管者、信任与安全团队、记者以及需要在纠纷中取证的人**来说，这种机制常常是**优点**：公开发言更接近现实世界——**在有见证的情况下说过的话，很难被说成「从未发生」**。内容一旦经过分发扩散，**抹除记忆（memory-holing）**的成本更高，有利于网络开放层上的**问责与可审计性**。（需要**强隐私**的用户，应把**私信与非公开渠道**视为另一条产品线——见下文。）

## 设计上的不足

大本营、分发与信息流的拆分，在架构上是实打实的优势：既给产品竞争留出空间，又让数据规则保持一致。但和所有雄心勃勃的基础设施选择一样，在**风险、成本与竞争格局**上也值得了解其代价。

### 纸面上的去中心化与日常体验

协议允许不止一家公司托管账号、运营分发或做信息流。实际上，用户与流量仍往往集中在少数大型托管方和知名应用上。这并不否定协议的可迁移性，但意味着**日常体验可能比「许多对等节点」的示意图看起来更集中**——类似电子邮件：原则上开放，但多数邮箱仍由少数服务商承载。

### 「人人都能看见公开帖」带来的规模与成本

公开内容被设计为流经共享的分发与索引。要在全球规模上跟上这股数据流，意味着任何想在「发现」中扮演核心角色的一方，都需要**巨大且持续增长的存储与带宽**——而不只是「给一个小社群搭台轻量服务器」。这会抬高新基础设施参与者的门槛，也使全球产品体验在一定程度上取决于**谁扛得起这种规模的运营成本**。

### 身份与信任

用户身份被设计成可迁移的，但部分省事的做法依赖**共享目录或常见的网络身份（域名）**。这对用户和开发者来说，比一上来每人自建一套密码学设施要容易，但也意味着**某些登记机构故障、DNS 问题或治理争议，可能影响账号解析是否顺畅**——这是易用性与减少对第三方依赖之间的常见取舍。

### 与 Mastodon 式「联邦宇宙」应用不是同一张网

Bluesky 的规则与较老的 ActivityPub 生态（例如 Mastodon）**不能即插即用**，需要单独的桥接产品与持续维护。这本身是产品与生态路径的选择，未必是孤立意义上的缺陷，但会影响**合作战略以及对「一个协议通吃」的预期**。

### 隐私预期与协议侧重点

按设计，公开发帖就是真正的公开。私信等功能，未必符合部分用户对「去中心化」品牌下的**端到端隐私**想象。做尽职调查时，宜把**私信与公开时间线**当作**两条不同的故事**，各自对应不同的风险画像。

### 发帖时间——协议不能保证什么

当人们默认帖文带有**经认证的、像公证时钟一样的时间戳**时，这一点关系到**合规、纠纷与品牌安全**。在朴素理解下，AT 协议并不是那样工作的：

- 记录上的**时间戳**大多由**发帖应用填写**（展示用元数据），而不是由网络独立「公证」为真实世界时间。规格里，那些**看起来像时间戳的标识符**仍被视为**用户可控**——适合排序与存储，**不能**当作内容创建时间的取证级证明。

应把**「发帖时间」**视为**产品层面的信任**，而非密码学意义上的证明——尤其在营销话术偏向「去中心化」时更要注意。

### 对更广泛开发者群体的复杂度

协议精确且能力强，因此相比「给另一台服务器发一条消息」这类更简单的模型，**正确实现的门槛更高**。环节越多，在工具链成熟之前，**第三方创新可能更慢**，或更容易出现细微实现偏差——在评估「基于标准的护城河」究竟能多快增厚时，这一点值得纳入考量。

## 借助自有链可以如何补强

「这件事到底发生在什么时候？」在 AT 协议里仍主要由主页服务器与客户端来呈现。**我们自有的区块链**可以在这一层之上增加一层**独立锚定**，而不必声称取代 AT 的角色分工。

- **承诺的是历史摘要，而非把整个帖文图搬上链。** 我们可以对用户发帖历史（或在发布过程中按约定生成的快照）做密码学哈希，将**摘要（digest）**以交易形式写入**我们**的链上。日后任何人都可以核对某段历史是否与当时的承诺一致。

- **把出块时间当作粗粒度、由共识背书的上界。** 我们链上的区块带有网络达成一致的时间戳。包含某一哈希的那一块，为该摘要建立了一个**上界**：系统在**不晚于**该块时间已知晓该摘要。出块间隔不是秒表，但界定了锚定写入发生的**大致时间窗口**——在纠纷叙事、合规说明或「某日期之前是否已存在？」一类问题上，比单一应用内的时间字段更有说服力。

- **如实界定能力边界。** 此举加强的是相对**我们账本**的**防篡改与排序依据**；并不能凭空公证作者设备时钟或中继上的每一份副本。在 AT 有意不把「经网络认证的墙钟时间」做重的前提下，这是补强**来源与存在性证明**的一条务实路径。

## 速读摘要

AT 协议在**角色边界清晰、数据可迁移、上层可做多款产品竞争**方面很强；在**公开层**上，**有利于问责的删除语义**（删除会更新官方图，但不会对所有观察者假装过去从未发生）也是一亮点。对投资人而言，主要问题包括：**全球分发是否仍可持续、可竞争**；网络变大后**身份与目录治理**是否仍受信任；**隐私定位**是否与用户预期一致；与其他开放社交技术栈的**生态隔离**是可接受的战略赌注还是合作上的逆风；以及相关方是否理解**发帖时间并非经网络认证的事实**。
